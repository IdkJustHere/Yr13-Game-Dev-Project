Game Design Document: Conveyor Kitchen

Platform: PC (Mouse & Keyboard) | Engine: Godot 4.x | Team: Solo Developer

Game Overview

Conveyor Kitchen is a spatial logistics puzzle game viewed from an isometric 3D perspective. The player designs efficient conveyor networks on a strict grid to transport food items—meat, fruit, and spices—from their source hoppers to designated preparation stations before the timer runs out.

The core tension arises from limited building resources and restricted floor space; every conveyor tile placed must be justified, as the player is graded not only on completing deliveries but on architectural elegance. The game rejects complex physics simulations in favor of discrete, deterministic movement: items snap from cell to cell in half-second intervals, transforming the experience into a hybrid of chess and factory planning.

Core Mechanics

Grid System & Placement

The game operates on a 1m × 1m grid system. The player surveys the layout from an isometric camera, noting the positions of colored Source nodes (which spawn items) and their matching Sink stations (which consume them). Using a limited inventory of conveyor tiles—typically ranging from twelve to twenty pieces depending on the challenge—the player must bridge the gap between them.

Placement is immediate and tactile. A ghosted grid overlay follows the mouse cursor, snapping to the nearest meter-square cell. Clicking deposits a straight conveyor section or a corner piece, which can be rotated in ninety-degree increments before finalizing.

Discrete Flow System

Items do not use physics simulation. Instead, they exist as discrete agents on the grid, advancing one cell every 0.5 seconds (the "tick"). This tick-based approach ensures deterministic, predictable movement essential for the puzzle nature of the game. Each tick, the system checks the next cell in the item's path: if it's empty, the item moves; if occupied, a Jam occurs.

Jam States & Crisis Management

If two items attempt to occupy the same cell simultaneously, the system enters a Jam State. The conveyor line halts, a visual indicator appears, and the player must manually intervene by clicking the jam location to clear it (incurring a 5-point penalty). Accumulating three simultaneous jams triggers a Kitchen Shutdown—immediate level failure.

Scoring & Medals

Success is measured across three tiers:

  • Bronze: Deliver the required quota before timeout
  • Silver: Complete with at least 30 seconds remaining
  • Gold: Complete using fewer than 50% of available tiles

Implementation Plan

This schedule assumes approximately 4 hours per week of focused development time.

Sprint 1: Foundation & Core Loop

Weeks 1–4 | Goal: Establish grid system, basic placement, discrete item movement, and camera controls using primitive meshes

Scene Specifications

game_manager.tscn (Node — Autoload)
Handles global state and level persistence.

Children: None (logic-only node)

Signals:
  • inventory_changed(count: int) → Emitted to ui_hud.tscn::_on_inventory_changed
  • game_state_changed(state: GameState) → Emitted to level.tscn::_on_state_changed (Planning vs. Rush Hour)
level.tscn (Node3D — Root gameplay scene)
Container for the playable space.

Children:
  • Camera3D: Orthogonal projection, positioned at isometric angle (45° Y, 35° X), script handles 90° snap rotation on mouse wheel
  • DirectionalLight3D: Shadows enabled (placeholder visuals need depth)
  • GridManager (Node3D): Owns the 2D array logic
  • TileGhost (MeshInstance3D): Semi-transparent white cube, follows mouse cursor via raycast
  • CanvasLayer (UI container): Instances ui_hud.tscn
grid_manager.tscn (Node3D)
The brain of the logistics system.

Children:
  • TickTimer (Timer): Wait time 0.5s, autostart false during Planning phase
Signals:
  • tick → Connected to all active Item nodes' _on_tick_move method
  • tile_placed(grid_pos: Vector2i, type: int) → Emitted to level.tscn for visual feedback
  • jam_detected(at_pos: Vector2i) → Future connection for Sprint 2
tile_straight.tscn (Node3D)
Placeholder straight conveyor.

Children:
  • MeshInstance3D: BoxMesh (1×1×0.2m), grey material
  • StaticBody3D: Collision for raycast detection
Script: Stores direction: Vector2i (e.g., (1,0) for East)
item_meat.tscn (Node3D)
Prototype for all items (recolored later).

Children:
  • MeshInstance3D: BoxMesh (0.4×0.4×0.4m), red material
Script: Handles position interpolation between grid cells; stores current_grid_pos and next_grid_pos
ui_hud.tscn (CanvasLayer)
Minimal interface for debugging.

Children:
  • TileCountLabel (Label): Displays remaining inventory
  • StateLabel (Label): Shows "PLANNING" or "RUSH HOUR"

Key Tasks

  • Configure Git repository with .gitignore for Godot
  • Implement raycast from Camera3D to horizontal plane (Y=0) to calculate grid coordinates
  • Create 10×10 grid data structure (2D array) in GridManager
  • LMB places tile_straight, decrements inventory counter
  • RMB removes tile, returns to inventory
  • Items spawn at (0,0) and move +1 X every TickTimer timeout
  • Camera rotation: Wheel input rotates Camera3D around look-at point in 90° increments

Appendix: Technical References

Appendix A: Grid Coordinate Conversion (Conceptual)

Reference for level.tscn camera script: Use Camera3D.project_ray_origin(mouse_pos) and project_ray_normal() to intersect with mathematical plane at Y=0. Convert resulting Vector3 to Vector2i by flooring X and Z coordinates. Account for camera rotation by ensuring raycast uses current camera transform.

Appendix B: Ghost Shader (Conceptual)

Spatial shader using render_mode unshaded, blend_mix. Uniforms: validity_color (green/red). Vertex shader: slight Y-offset oscillation using TIME. Fragment shader: sample albedo texture if present, multiply by validity_color with transparency 0.6. Discard pixels below alpha threshold for cutout effect.

Appendix C: Audio Implementation Pattern (Conceptual)

AudioManager singleton maintains array of AudioStreamPlayer nodes. Method play_sfx(stream: AudioStream) finds first non-playing player, assigns stream, and plays. Prevents rapid-fire sound cutoffs (e.g., multiple placement clicks). Background music uses separate AudioStreamPlayer with loop enabled.
